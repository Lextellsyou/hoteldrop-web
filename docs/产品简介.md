# 今夜有房（Hotel Drop）产品简介

> **文档性质：** 项目级产品概览，面向产品、设计、研发与协作方快速了解「这是什么、解决什么问题、怎么运转」  
> **对照实现：** 当前 H5 Demo（`hoteldrop-user-0.2` + `hoteldrop-store`）  
> **更新日期：** 2026-06-07

---

## 一、一句话介绍

**今夜有房** 是一个面向 **当天临期酒店库存** 的撮合平台：用户用更短的路径（位置 + 价格 + 房型偏好）筛出高性价比房间并下单；酒店将尾房以可控方式售出，**不破坏 OTA 公开价格体系**。

它不是传统 OTA（先选酒店、再比价），而是 **「先定需求，再选具体房间」** 的尾房捡漏模式。

---

## 二、为什么做这件事

### 供给侧

- 酒店每天存在大量 **临期未售库存**（尤其当晚房），房租与人力成本固定，空房即损失。
- OTA 不适合大幅降价，容易冲击渠道价盘与品牌定价。
- 旅行社、地接等也存在包房库存，卖不掉即亏损。

### 用户侧

- 用户想要 **更低价格**，但不愿在多个 App 里反复比价。
- 对「在可控条件下接受一定随机性」有接受空间——用 **不确定性换性价比**。

### 平台机会

在 OTA 与线下履约之间做 **轻撮合**：线上收信息服务费确认意向，**房费大额在线下酒店前台完成**，平台聚焦库存匹配与履约协同，而非代收代付全款。

## 三、产品定位

| 维度 | 今夜有房 | 传统 OTA（携程等） |
|------|----------|-------------------|
| 决策路径 | 位置 + 预算 → 筛出 3 个房间 → 选定下单 | 选城市/酒店/房型 → 比价 → 下单 |
| 库存类型 | **当天临期全日房**（尾房） | 全量可售库存 |
| 价格逻辑 | 尾房特价，通常低于 OTA 公开价 | 明标价，促销有限 |
| 平台角色 | 库存撮合 + 信息服务 | 展示、交易、客服一体化 |
| 资金结构 | 线上信息服务费 + 到店付 **全额房费** | 线上预付全款或担保 |

**放弃用户：** 必须指定某一家酒店、强计划型商务差旅人群。

**目标用户：** 价格敏感型年轻人、追求性价比的轻度品质用户、愿意尝鲜的盲盒/捡漏型用户。

## 四、核心用户旅程（当前 Demo）

### 4.1 用户端

```text
首页「我要订房」
  → 基础信息（城市 / 地图选点 + 半径 / 价格档位 / 大床·双床 / 有窗·无窗）
  → 「开始筛选」
  → 房间列表（每批最多 3 张房间卡：酒店名、房型、窗型、价格等）
  → 可选「换一批」（前 2 次免费，第 3 次起 ¥9.9/1 次 或 ¥19.9/3 次）
  → 选定房间 → 确认订单
  → 线上支付信息服务费（房费 × 10%）→ 订单确认，展示酒店信息与订单码
  → 到店：出示订单码 + 报手机尾号 → 酒店核验
  → 酒店前台支付全额房费 → 入住 → 离店评价
```

**关键规则：**

- 仅 **当天全日房**，固定「今晚入住」，限 **2 人**，不做钟点房。
- 合作酒店每天 **16:00** 放出当日可售尾房库存。
- 支付信息服务费后 **立即锁库存、订单生效**（无需酒店二次抢单确认）。
- 订单确认后 **15 分钟内** 可取消并退还信息服务费；超时不可退。
- **换批筛选费**（¥9.9 / ¥19.9）为虚拟服务，任何阶段不退。

### 4.2 酒店端

```text
登录门店账号
  → 尾房库：维护 4 种固定 SKU（大床/双床 × 有窗/无窗）的价格、原价、库存
  → 每天 16:00 库存对用户侧开放
  → 工作台：查看今日订单
  → 到店核验：输入订单码，核对用户手机尾号后四位
  → 核验成功后提示前台收取全额房费
  → 标记「已收取房费」→ 订单完成
```

**关键规则：**

- 酒店账号由 **平台后台开通**，无自助注册。
- 已取消「抢单 / 待确认接单」链路：用户付服务费即成交，酒店侧以 **履约与核验** 为主。
- 住宿发票由 **酒店** 开具；平台服务费 / 筛选费由平台侧合规开票（正式版规划）。

## 五、商业模式（MVP 口径）

| 收入项 | 说明 | Demo 实现 |
|--------|------|-----------|
| **订单信息服务费** | 按房费 **10%** 收取，用户线上支付以确认订单 | `INFO_SERVICE_FEE_RATE = 0.1` |
| **换批筛选费** | 同一需求内第 3 次起付费筛选，¥9.9/1 次、¥19.9/3 次 | 用户端筛选弹窗 |
| **到店房费** | 用户在前台向酒店支付 **全额房费**，不计入平台线上实收 | 酒店端收款提醒 |

**示例（房费 ¥288/晚）：**

- 线上信息服务费：¥29（10%）
- 到店房费：¥288（全额）
- 用户合计支出：¥317

信息服务费 **按房费计价，但与房费账单分离**；不是「线上 10% + 到店 90% = 100% 房费」的拆分支付。

## 六、双端协作与订单状态

```text
用户付服务费 → 待使用（已锁库存、有订单码）
      ↓
酒店核验手机尾号 → 已核验（提示收房费）
      ↓
酒店确认已收房费 → 已完成
      ↓
用户待评价 → 已评价
```

双端 Demo 通过浏览器 **同源 `localStorage`** 同步订单（须同一 `localhost` 端口）。

## 七、本仓库是什么

本仓库是 **今夜有房 MVP 的静态 H5 Demo**，用于验证产品流程、UI 还原与双端联调，**非生产环境**。

| 目录 | 说明 |
|------|------|
| `hoteldrop-user-0.2/` | 用户端 H5（首页 / 筛房 / 下单 / 订单 / 我的） |
| `hoteldrop-store/` | 酒店端 H5（工作台 / 尾房库 / 核验 / 我的） |
| `design-rules/` | UI 设计交付包（Token、组件、页面对照、小程序映射） |
| `docs/` | 项目级产品说明与双端对齐文档 |
| `hoteldrop-user-0.2/docs/` | 用户端 PRD、需求清单、会议纪要 |
| `hoteldrop-store/docs/` | 酒店端 PRD、改造说明 |

## 十、愿景一句话

**用更短的决策路径，帮用户捡到当晚尾房好价；帮酒店清掉临期库存，又不公开破价。**
