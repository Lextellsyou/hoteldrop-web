# 双端产品修改说明（0601 讨论对齐）

> **文档性质：** 基于 `hoteldrop-store/docs/盲盒问题讨论+0601.pdf` 与产品侧整理，对 **用户端**（`hoteldrop-user-0.2`）与 **酒店端**（`hoteldrop-store`）的修改方向统一说明  
> **对照基线：** 当前 Demo 代码、`hoteldrop-user-0.2/docs/用户端0.2-需求调整清单.md`（2026-05-25 定稿）、`hoteldrop-store/docs/酒店端产品需求文档.md`（V1.1）  
> **日期：** 2026-06-01  
> **状态：** 待评审 → 评审通过后分拆进各端 PRD 与开发任务

## 零、核心决策变更（相对旧 PRD）

本次讨论对撮合模型做了 **根本性简化**，以下结论 **覆盖** 旧文档中「抢单 / 酒店确认 / 匹配截止 / 多轮广播」等相关描述。

| # | 旧模型 | 新模型（0601 定稿方向） |
| --- | --- | --- |
| 1 | 用户付定金 → 进入「待确认」→ 酒店抢单/确认接单 → 解锁信息 | 用户付定金 → **库存已放出则直接成交**，无需酒店二次确认 |
| 2 | 酒店需在工作台响应需求单（30s～2min SLA） | **取消抢单逻辑**；酒店侧不再出现「待响应需求单 / 确认接单」 |
| 3 | 尾房随时新建、字段多、逐条录入 | **固定 4 种 SKU**（床型 × 窗型），默认创建后在列表中改 **价格 / 原价 / 库存** |
| 4 | 入住日可选、表单字段多 | **仅当日库存**；日期固定当天，表单不再选日期 |
| 5 | 酒店登录：手机号 + 短信验证码 | **密码登录**；注册不走验证码，**微信拉起本机手机号** 作为登录凭证 |
| 6 | 酒店自助注册 | **无独立注册入口**；账号由 **平台后台开通**；「我要加盟」仅留资 |
| 7 | 库存随时上架 | 酒店 **每天固定 16:00** 放出当日可售数据（定时释放，非实时抢单触发） |

**一句话：** 酒店提前维护当日 4  SKU 库存并在固定时点放出；用户选房间付定金即锁库存，平台不再走「发单—抢单—确认」链路。

## 三、酒店端修改规格

### 3.1 信息架构（目标）

```text
底部 Tab（建议 3 个）
├── 工作台（首页，默认 landing）
│     ├── 顶栏：当前酒店名 + 【切换酒店】（多店账号时）
│     ├── 指标卡 × 2：可售尾房 · 今日订单
│     └── 今日订单列表（本店、当日）
├── 尾房库
│     └── 4 条固定 SKU 列表 → 点击进入编辑（价格/原价/库存/上下架）
└── 核验（或合并进工作台订单详情，二选一 MVP）
      └── 到店扫码/输入订单码核验（保留现有能力）

我的 / 账号
└── 从顶栏入口进入：账号信息、门店资料、退出登录等
```

> **与旧 PRD V1.1 差异：** 取消「工作台 + 待响应需求单 + 抢单详情」；工作台变为 **经营看板 + 今日订单**，不是撮合大厅。

### 3.2 切换酒店入口

| 项 | 定稿 |
| --- | --- |
| 位置 | **顶栏左侧或标题区**，与酒店名同级；多店账号必显，单店可隐藏 |
| 交互 | 点击 → Bottom Sheet / 下拉列表，展示有权限的门店 |
| 切换后 | **工作台指标、尾房库 4 SKU、订单列表** 全部按 `storeId` 刷新 |
| 与旧方案 | 不再放在「我的」深处或仅 Demo 调试入口 |

### 3.3 工作台首页

#### 3.3.1 指标卡（仅 2 个）

| 卡片 | 口径 | 示例 |
| --- | --- | --- |
| **可售尾房** | 当前门店、当日、状态=上架 且 库存>0 的 SKU **份数合计** | `12 间` |
| **今日订单** | 当前门店、入住日=今日 的订单数（含待核验+已核验） | `5 单` |

**移除：** 待响应需求数、履约中订单数（若与「今日订单」重复则合并口径）、撮合说明卡片、抢单相关任何模块。

#### 3.3.2 今日订单列表（须补原型字段）

列表为空时的占位文案需引导去尾房库维护库存；有数据时每行建议展示：

| 字段 | 说明 | 必填 |
| --- | --- | --- |
| 订单码 | 如 `HD-20260601-001`，等宽字体 | 是 |
| 下单时间 | 当日 HH:mm | 是 |
| 房型 SKU | 如「大床 · 有窗」 | 是 |
| 卖价/晚 | 成交金额 | 是 |
| 客人称呼 | 脱敏，如「张* · 尾号 6281」 | 是 |
| 核验状态 | 待核验 / 已核验 | 是 |
| 操作 | 点击进入核验详情 | 是 |

**列表筛选（可选 MVP）：** 全部 / 待核验 / 已核验。

**行点击：** 进入核验详情（复用现有 `#/verify/:id` 能力）。

### 3.4 尾房库 — 固定 4 SKU 模型

#### 3.4.1 SKU 定义（与用户端对齐）

| SKU ID | 展示名 | 对应用户端字段 |
| --- | --- | --- |
| `king_with` | 大床 · 有窗 | `bedType: king`, `windowPref: with` |
| `king_without` | 大床 · 无窗 | `bedType: king`, `windowPref: without` |
| `twin_with` | 双床 · 有窗 | `bedType: twin`, `windowPref: with` |
| `twin_without` | 双床 · 无窗 | `bedType: twin`, `windowPref: without` |

#### 3.4.2 默认创建逻辑

- 门店 **首次激活** 或 **每日 0 点/首次进入尾房库** 时，若当日无记录 → **自动创建 4 条 SKU**（库存默认 0 或上次模板值，产品可配置）。
- **取消「新建尾房」FAB 主路径**；列表始终展示 4 行，运营直接点行编辑。
- FAB 可改为「批量设置」或移除。

#### 3.4.3 编辑字段（精简后）

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 房型 SKU | 只读 | 四选一，不可新增类型 |
| 卖价 / 晚 | 是 | 对用户展示的成交价上限参考 |
| 原价 / 晚 | 是 | 用于用户端「省了多少钱」展示 |
| 库存数量 | 是 | 整数 ≥ 0 |
| 状态 | 是 | 上架 / 停用（售罄可由库存=0 推导或单独标记） |

**去掉：** 适用入住日（固定当天）、可入住时段、偏好标签、最大人数（固定 2 人）、连住晚数（固定 1）、窗型单独选项。

#### 3.4.4 列表行展示（尾房库）

| 字段 | 说明 |
| --- | --- |
| SKU 名 | 大床 · 有窗 |
| 卖价 / 原价 | `¥268/晚 · 原价 ¥399` |
| 库存 | `库存 3` |
| 状态 | 上架 / 停用 / 已售罄 |

### 3.5 每日 16:00 库存释放规则

| 项 | 定稿 |
| --- | --- |
| 释放时点 | 每天 **16:00**（可配置时区，默认酒店所在城市） |
| 含义 | 16:00 前用户端 **不可见/不可订** 该酒店当日尾房；16:00 后按尾房库中已维护的 SKU 与库存进入可售池 |
| 酒店操作 | 16:00 **前** 在尾房库编辑好 4 SKU 的价格与库存；**无需** 额外点击「确认放出」 |
| 16:00 后修改 | 允许改价/改库存，但需定义对已下单的影响（MVP：**仅影响后续新单**，已锁库存不回溯） |
| Demo | 提供「模拟：已到 16:00」开关或系统时间判断 |

### 3.6 登录与账号

#### 3.6.1 登录

| 项 | 定稿 |
| --- | --- |
| 主方式 | **手机号 + 密码** |
| 快捷方式 | **微信授权 → 拉起本机手机号** → 若手机号已绑定账号则直接登录 |
| 去掉 | 短信验证码输入框、「发送验证码」、任意 4 位 Demo 验证码 |

#### 3.6.2 注册 / 入驻

| 项 | 定稿 |
| --- | --- |
| 酒店自助注册 | **不做**。账号由 **平台运营后台** 创建并绑定门店 |
| 「我要加盟」 | 保留为 **留资页**（酒店名称 + 联系电话），提交后商务跟进，**不自动开通** |
| 安全说明 | 若未来开放「仅填酒店名+手机号」免验证注册，需单独评估；**本期不做** |

#### 3.6.3 页面路由（调整）

| 路由 | 说明 |
| --- | --- |
| `#/login` | 密码登录 + 微信手机号快捷登录 |
| `#/join` | 加盟留资（不变） |
| `#/` 或 `#/dashboard` | 工作台首页（登录后默认） |
| `#/inventory` | 尾房库（4 SKU 列表） |
| `#/inventory/edit/:skuId` | 编辑单 SKU |
| `#/verify` | 核验列表（可与工作台合并，见 3.1） |

## 四、用户端修改规格

### 4.1 订单状态机（简化）

```text
旧：
  付定金 → 待确认（等酒店）→ 待使用 → 待评价 / 预订失败

新：
  付定金 → 待使用（立即锁库存、展示酒店信息与订单码）
         → 15 分钟反悔期内可取消退定金
         → 超过 15 分钟不可退
         → 到店核验 → 待评价 → 已评价

失败分支（仅保留）：
  库存不足 / 支付时校验失败 → 预订失败（定金不退或原路退，按支付结果）
  用户 15 分钟内主动取消 → 预订失败（failReason: user_cancel，退定金）
```

**移除状态：** `待确认`（或保留枚举但不再使用，迁移为 `待使用`）。

**移除 UI：** 订单详情「等待酒店确认」卡片、Demo「模拟酒店已接单/无法接单」按钮。

### 4.2 付定金成功后的行为

| 项 | 定稿 |
| --- | --- |
| 库存校验 | 支付前/支付时校验：目标 SKU 库存 > 0 且 当前时间 ≥ 当日 16:00 且 酒店 active |
| 锁库存 | 支付成功 **原子扣减** 对应 SKU 库存 |
| 订单字段 | 写入 `hotelConfirmedAt` = 支付成功时刻（或改名为 `lockedAt`）；`refundDeadlineAt` = 支付时刻 + 15 分钟 |
| 展示 | 订单详情 **立即展示** 酒店名、地址、订单码、导航信息 |
| Toast/文案 | 「定金支付成功，订单已确认」 |

### 4.3 退款规则（修订）

| 阶段 | 规则 |
| --- | --- |
| 付定金后 15 分钟内 | 用户可「取消订单并退定金」；筛选服务费不退 |
| 超过 15 分钟 | 不可退款；进入正常履约 |
| 酒店拒单 | **不再存在**（无酒店确认环节） |
| 库存不足 | 支付前应拦截；若并发导致超卖，平台侧退款并提示「库存已被抢完」 |

**订单详情 UI（待使用 + 在反悔期内）：** 保留现有倒计时与「取消订单并退款」按钮，文案改为「订单确认后 15 分钟内可取消」。

### 4.4 定金页与二次确认文案

**需替换的关键句：**

| 位置 | 旧文案方向 | 新文案方向 |
| --- | --- | --- |
| 定金页警示 | 付定金后等待酒店确认 | 付定金即锁定房间；**15 分钟内** 可取消退定金 |
| 二次确认弹窗 | 酒店确认后 15 分钟… | **订单确认后 15 分钟** 内可取消并退定金 |
| FAQ | 待确认、酒店接单 | 16:00 后开放当日尾房；付定金即确认 |

### 4.5 筛选与匹配（与用户端 0.2 继承关系）

以下 **保持不变**（见 `用户端0.2-需求调整清单.md`）：

- 仅地图选点 + 半径；价格区间；大床/双床 + 有窗/无窗
- 当天全日房、限 2 人
- 3 张房间卡、筛选次数付费
- 微信登录 + 手机号姓名；无短信验证码

**新增硬条件：**

- 仅召回 **当日 16:00 已释放** 且库存 > 0 的 SKU
- 床型必须一致；窗型优先一致（不足 3 条时的降级规则不变）

### 4.6 用户端页面结构（无大改）

```text
Tab：首页 / 订单 / 我的
  订单列表：移除「待确认」筛选项，或合并进「待使用」
  订单详情：按 4.1 新状态机展示
```
